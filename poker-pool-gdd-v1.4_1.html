Executive Summary
One-Line Pitch

Pocket pool balls to build a winning poker hand — assign suits to holes as you play, and win the showdown.

Poker Pool is a 1v1 game on a standard 6-pocket billiards table. Players alternate pool shots, pocketing balls (ranked Ace–King) into holes. Holes get assigned poker suits during play. Every pocketed ball registers as a playing card. Race to build the strongest 5-card poker hand — both hands always visible, strategy is pure.

15Balls
6Pockets
4Suit holes
2Wild holes
5Cards / hand
3Misses = loss

What Makes This Different

  • Suit assignment during play. No suits are pre-set. The first player to pocket a ball into any hole names that hole's suit (♠ ♥ ♦ ♣). Once all four suits are assigned, the two remaining holes become Wild Holes for wildcard balls. Suits themselves carry no ranking advantage in poker — this mechanic is about choosing which pocket to aim at, not which suit is superior.
  • Full information. Both hands always visible. No bluffing — pure shot-making and card strategy.
  • Break is exciting, not decisive. Balls pocketed on the break respawn — no free cards from a lucky scatter.
  • Card-decorated balls. Each ball displays its rank (A, 2–10, J, Q, K), bridging both game worlds visually.
  • Live table. Every pocketed rank ball spawns a fresh copy, keeping the table in play throughout.

Table, Balls & Setup

The Table

Standard 6-pocket pool table (2:1 ratio). Four corner pockets plus two centered side pockets. All pockets start unlabeled at game start.

Rack Formation

Standard triangle rack at the foot spot. Balls 1 (A), 11 (J), 12 (Q), 13 (K) are fixed in the 4 center positions of the triangle, protected from a lucky break scatter. All other balls (2–10, 14, 15) fill remaining positions randomly. The apex ball must not be a center-cluster ball.

The Lag & Break Shot

Lag (Who Breaks)

Both players simultaneously shoot a ball from behind the head string toward the foot rail. The ball returning closest to the head rail without touching it wins. The winner chooses whether to break or pass. If both touch the rail or pocket, re-lag.

Demo Shortcut

Replace lag with a coin-flip animation for the prototype. Implement the real lag post-demo.

Break Shot

Legal break: ≥4 balls contact a rail, OR ≥1 ball pocketed. If illegal, opponent chooses accept-or-re-rack.

Break OutcomeResult
0 balls pocketedTurn lost. Opponent begins their regular turn.
1+ balls pocketedAll pocketed balls respawn. No cards registered. No suits assigned. Breaker keeps turn and shoots again under normal rules.
Cue ball scratch on breakAll other pocketed balls respawn. Turn lost (+1 miss). Opponent gets Ball in Hand anywhere on table.
4
Suit Assignment

All 6 pockets start unlabeled. The first 4 separate holes to receive a ball become Suit Holes. The player who pockets the ball names that hole's suit (♠ ♥ ♦ ♣). Each suit is assigned to exactly one hole, permanently, for the whole game. No contesting.

Suit assignment is a positional decision — it determines which pocket builds which suit, letting both players plan routes to Flushes or Straights. In poker, all four suits carry equal rank; assigning a suit is purely about controlling pocket-to-suit geography on the table.

Once all 4 suits are assigned, the 2 remaining holes become Wild Holes (gold rim + ★). Only balls 14–15 register in Wild Holes; all other balls pocketed there respawn.

Pocket StateVisualBehavior
UnclaimedGray rimFirst ball pocketed triggers suit-naming UI
Suit Hole (♠/♥/♦/♣)Colored rim + suit glyphPocketed rank balls register as cards of this suit
Wild HoleGold rim + ★Balls 14–15 register here; others respawn

The aiming assist overlays validity glows on pockets while aiming (see Section 12) — players always know at a glance which pocket is valid for the ball they're targeting.

Duplicate prevention: A player cannot hold two cards with identical rank AND suit. The aiming assist derives validity live from current hand state — no pre-stored flags needed.

Wildcards (Balls 14–15)

  1. Ball pocketed into a Wild Hole
  2. Player selects any rank (duplicates greyed out)
  3. Player selects any of the 4 assigned suits
  4. Card registered. Ball permanently removed (does not respawn).
Wildcard Availability

Wild Holes only open after all 4 suits are assigned. Balls 14–15 pocketed into Suit Holes before that respawn — they cannot register early.

6
Turn Structure

Players alternate turns. The non-breaking player goes first after the break. One shot per turn.

Turn Flow

  1. Aim: position cue, set spin contact point, set power
  2. Shoot: release cue
  3. Physics resolves
  4. Outcome: valid pocket → card registered + bonus shot available; no valid pocket → turn ends, miss +1

Bonus Shot

Pocketing any ball in a valid hole grants one immediate bonus shot. Chain continues as long as valid pockets are made. Chain ends on a miss or voluntary end-of-turn.

Scratch (Cue Ball Pocketed)

All co-pocketed balls respawn; no cards registered. Turn ends, miss +1. Opponent gets Ball in Hand (cue ball placed anywhere on table).

OutcomeMiss CounterTurn PassesBonus Shot

Hand Building

Each player holds up to 5 cards. Both hands always visible to both players.

Hand SizeOn Next Pocket
0–4 cardsNew card added automatically
5 cards (full)Swap Decision: player chooses which of the 6 cards (5 existing + new) to discard. Discarded card gone permanently. Hand stays at exactly 5.

Choosing to discard the new card is valid — useful when the new card doesn't improve the hand but potting the ball was tactically important (e.g., denying a good position to the opponent).

3-Miss Elimination

3 consecutive failed turns (miss / invalid pocket only / scratch) = immediate loss. Counter resets on any valid pocket. Prevents stalling; all shots must attempt a genuine pocket.

HUD Requirements

  • Both card hands always visible (Kenney Playing Cards Pack sprites)
  • Best hand rank label below each hand, updated in real time via pokersolver
  • 3-pip miss counter per player (● miss / ○ available) near player name
  • Stand button active only when player holds exactly 5 cards and it is their turn

Stand Timing

Declaring Stand forces a 3-turn clock on your opponent. But every turn you delay continuing to improve your hand is also a turn they get. Stand too early with a weak hand; wait too long and they finish first.

Cue & Shot Controls

The Visible Cue

A full-length pool cue is rendered on the table at all times during the active player's turn. The cue visually connects from behind the cue ball, extending outward in the opposite direction of the aimed shot. As the player adjusts aim, the cue rotates in real time around the cue ball as its pivot point, always pointing away from the target direction.

Cue Visual Spec

  • Length: extends from the cue ball tip to beyond the table edge — visible even at full zoom-out
  • Taper: thin at the tip (touching cue ball), thicker toward the butt end — matches a real cue silhouette
  • Stroke animation: when the player shoots, the cue slides forward through the cue ball in a quick thrust (0.15s forward, invisible snap-back). The cue disappears after the shot and reappears when the table has settled and the next turn begins
  • Power indicator: the distance the cue is "pulled back" from the cue ball visually represents shot power. At zero power, the tip just touches the ball. At max power, the cue is pulled back ~40% of its length from the ball before the stroke

Physics Requirements (MVP)

Required: ball–ball elastic collision, ball–rail bounce, cue ball spin/english (top / back / side-spin), shot power control via cue pull-back distance, ball friction and deceleration. Not in MVP: jump shots, masse shots.

12
Aiming Assist

Always active while aiming. Shows pocket validity for the currently targeted ball. Never suggests the best strategic move — the decision stays with the player.

  • Trajectory line: dotted line along the cue's aim axis showing cue ball path + first-struck ball path. Terminates at first obstruction or pocket. Spin shots are approximate (full spin prediction is a post-MVP feature).
  • Pocket glows: Green (valid card), Red (ball would respawn), Yellow pulse (unclaimed → naming UI will open), Gold star (Wild Hole).
  • Multi-ball chains not shown — preserves skill ceiling.
13
Respawn System

Triggers: Ball 1–13 enters Wild Hole · Ball 14–15 enters any Suit Hole · Any ball pocketed during a scratch · Any ball pocketed on the break shot.

Sequence: Ball vanishes from pocket → random available respawn spot selected → brief drop-in animation (~0.3s) → ball re-enters play with original rank. Validity is derived live from game state on each aim event — no suit restriction flags stored on individual balls.

Config Architecture

All tunable values — numbers, colors, asset paths, and audio files — live in three plain config files. Code never hardcodes anything. The AI agent generates full inline documentation for every variable when it creates each file.

Three Config Files

FileAudienceContains
config/gameplay.tsGame designer / producerPhysics constants, timing, scoring rules, hand limits, miss count, respawn coordinates, wildcard IDs
config/ui.tsDesigner / non-techColors, sizes, font sizes, animation durations, opacity, pocket glow colors, card dimensions, audio volumes
config/assets.tsDesigner / artistAll asset file paths — textures, sprites, audio files. Swap any asset by changing one string.

Live-Tweakable vs File-Only Variables

Not all variables should appear in the live dev panel. Variables that change fundamental game structure require a full reload anyway and would confuse non-technical users if surfaced at runtime. The distinction:

CategoryWhere to changeExamples
Live-tweakable — safe to adjust during a session, takes effect instantlyDev panel slider/picker AND config fileBall restitution, friction, shot power, all colors, animation durations, trajectory opacity, pocket glow colors, audio volumes, cue dimensions
File-only — structural or game-breaking if changed mid-session; requires page reloadConfig file only (not shown in dev panel)Total number of balls, wildcard ball IDs, hand size, suit hole count, rack positions, respawn spot coordinates, physics world gravity
Rule for AI Agent

Every variable in all three config files must be tagged in its JSDoc as either @live (appears in dev panel) or @fileonly (config file only, not surfaced in UI). The dev panel reads only @live variables and auto-generates the appropriate control (slider for numbers, color picker for hex strings, toggle for booleans, text field for paths).

Swapping Assets

Any asset can be replaced by dropping a new file into the assets/ folder and updating the corresponding path in config/assets.ts. The game loads assets dynamically at startup from these paths — no code changes, no rebuild required for a simple asset swap. Rebuild (Vite) is only needed when changing TypeScript logic.

Live Dev Panel — Prototype Layout

During prototyping the game runs in a split-screen developer layout: the game canvas occupies the left two-thirds of the screen, and the right one-third is a live configuration panel. The panel is part of the prototype build only — it is stripped out for the final deployed game.

Layout Spec

AreaWidthContents
Game canvas66% of viewportFull Poker Pool game, scaled to fit
Dev panel33% of viewportLive controls for all @live config variables

Dev Panel Behavior

  • Auto-generated from config: the panel reads all variables tagged @live in gameplay.ts and ui.ts and renders an appropriate control for each — slider for numbers, color picker for hex strings, toggle for booleans. No manual panel wiring needed.
  • Grouped by section: panel controls are grouped under the same section headings used in the config files (PHYSICS, COLORS, CUE, ANIMATION, AUDIO, etc.), collapsed by default, expandable.
  • Instant effect: changing a value in the panel immediately updates the live game state. Physics values (restitution, friction, power) apply on the next shot. Visual values (colors, sizes) apply in the next render frame.
  • Export button: a small "Copy to config" button at the bottom of the panel prints the current panel values as a ready-to-paste block for gameplay.ts / ui.ts. This lets a non-technical team member tweak visually and hand the output to a developer to persist.
  • Reset button: reverts all panel values to the defaults defined in the config files.
  • Panel hidden in production: the split-screen layout is enabled only when the URL contains ?dev=1 or when NODE_ENV !== 'production'. In the deployed build the panel is invisible and the game canvas fills 100% of the viewport.

Guardrail for AI Agent — Dev Panel

When building the dev panel (Milestone 1), the AI must: (1) parse the @live JSDoc tags from config files to auto-generate controls — do not hardcode which variables appear in the panel; (2) wire each control so changes call a setConfig(key, value) function that updates a runtime config store and emits a 'configChanged' event that all game systems listen to; (3) ensure the panel is entirely absent in the production Vite build (tree-shaken via the ?dev=1 condition); (4) add a visible label on the panel reading "PROTOTYPE DEV MODE — not visible to players".

Layout: - The entire page is split into a flex row: left 66.67% = game canvas wrapper, right 33.33% = panel. - Panel: dark background (#1a1a1a), white text, fixed height 100vh, overflow-y scroll. - Panel header: "⚙ POKER POOL — DEV MODE" in red. Subtext: "Not visible to players." Auto-generate controls: - Read ALL keys from getCurrentConfig() (these are the @live variables). - For each key, infer the control type from the value: - string starting with '#' → - number between 0 and 1 (inclusive) → + number display - number > 1 and ≤ 1000 (px or ms values) → + number display - boolean → styled as toggle - small integer (≤ 20) → stepper buttons (+/−) + number display - Group controls under collapsible section headers matching config file sections (PHYSICS, SHOT, SCORING, COLORS, CUE, ANIMATION, AUDIO). Collapsed by default. - Each control: label (variable name formatted readable, e.g. BALL_RESTITUTION → "Ball Restitution"), control widget, current value display. - On any change: call setConfig(key, newValue).
